Problem Domain and Root Concept

In the past year or so I have become increasingly interested in back end authorization and web scraping overall. Essentially, the delivery of viable data to the front-end user which would enable the user to make educated decisions on whatever subject matter there is at hand. For back-end authorization, I became interested in APIs with OAuth 2.0, as I was able to understand the general back-end flow, and many of the services that offer APIs use OAuth 2.0. For this project, I wanted to identify a problem that I would be able to fulfill (or at least aid in solving) for a wide array of users.

(Etsy API Reference)

When looking at APIs, I came across the Etsy API which as a multitude of functions that allow sellers to manipulate their storefronts and data. The question I had was: is there enough sellers to really make something worthwhile? It turns out that Etsy has a very large user base and there are millions of sellers on this platform. Therefore, I wanted to take the seller perspective and see what I had to offer this group of stakeholders which is really a proxy to buyers and the company overall. The overall concept of my system (in a broad sense) would be to provide a viable solution in order to help Etsy sellers increase their overall sales.

When looking at how to do this, I stumbled across some forums that had questions revolving around the importance of listing elements known as “tags”. It turns out that tags are keywords that sellers are able to feed the Etsy search algorithm in order to help it place listings when a customer searches. Although these are not the end all be all in terms of listing views and sales, I concluded that finding the best and most relevant tags for a given keyword would be best.

The root concept was to take user input for a given keyword and have the system go find the very best tags that match the keyword. Therefore, the user would have the best chance at placing their listing at the top of the results when a potential buyer searched that keyword. This would address the overarching problem of increasing sales and cover the sub-problems of how to get more views as well as how to get the best tags.

Requirements Analysis

The first major part of my requirements analysis and field studies were the overall functionality of the back end. This is a crucial part to the entirety of the system since there is minimal effort or reliance on the front end. First, I had to determine how to authorize the server to pull data from other listings and see what data was available through the API. After consulting the Etsy API documentation, I found a few key results that would go on to really shape my system. First, I was able to find that the Etsy API has various objects. The main object that I was focused on was the listing object, which turned out to have an array element that housed tags and a function to pull listings by keyword.

(Etsy API Reference Tags)

(Etsy API Reference Pull Listings By Keyword)

At this point, I had discovered a major part of my system’s intended functionality. I would be able to take a keyword from the user and return arrays of keywords that matched the top listings. However, I next discovered Etsy’s rate limits.

(Etsy API Rate Limits)

It turned out that Etsy offered 10,000 public API calls per user per day. Therefore, I could only make 10,000 keyword calls per day. I had to make the decision whether or not I would be able to handle this with just my own account, or if I would have to authenticate users and use their pulls. There were various issues on both sides with the former being whether or not I would run out of pulls in a given day. The ladder rested on the fact that there could be privacy concerns with users needing to login and give me access to make requests on their behalf. I chose to use my own account because Etsy allows a developer to make more pulls upon request. I ultimately concluded that if my system were doing more than 10,000 pulls per day, then I could easily make that request.

The next major part of the exploration was how to authorize myself with and obtain a persistent token. As I previously mentioned, I understood how to authorize with OAuth 2.0 (which Etsy uses), but implementations are based on the developer, with the protocol being more of a base. Once I was able to get permanent credentials, I would be able to make authorized requests when prompted by a user, and return them in a viable format so that the user could perceive, interpret, and make sense of what was in front of them (gulf of evaluation) and then go on to the gul of execution. After experimenting with the API, I was able to produce the result that I hoped to receive from the server in a test environment that could be easily changed over to production. At this point, it was time to start designing the UI and determining how to form the UX.

User Interface Design

In order to design my user interface, I used a number of key concepts from Scenario Based Design (SBD) in order to take the user and broader stakeholder groups into account. I first started by using Hierarchical Task Analysis (HTA) in order to decompose the various tasks that the user had to go through in order to achieve the goal of obtaining the keywords. My breakdown is represented as follows.

(HTA)

This diagram helped me to map out the user flow of my intended system so that I could use it as a guide for eventual implementation. This is what I used to spur ideas, but I used other SBD principles and concepts in order to facilitate further planning. The second method I used was evaluating based on the principles within information design. This would help me to design and arrange the visual aspects of my UX(UI). This involved the gulf of evaluation given that I would need to hit all the points of perception, interpretation, and making sense. More specifically, I would have to make sure that the user is able to understand the base structure of my application with structures like lines, shading, and more (perception). I would also have to make sure that users would understand what the structures meant. For example, they would need to know that the search bar is used to search the keyword (interpretation). Finally, I would need to make sure that the output made sense in terms of the linkage between the input and output (keyword in, tags out, making sense).

(Prototype)

The final prototype uses numerous gestalt principles including closure, similarity, and area to help with perception. For example, the search bar (which has common understanding as a virtual metaphor), uses the principle of area. It is clear that the explicit border structure is used to emphasize the inside of the bar to prompt the user to enter information. Each of the three results uses more explicit structure to emphasize individuality while not over complicating the UI by using similarity across all three of the elements. This was the result of multiple prototypes and design critiques from test subjects.

Activity design is a major component of SBD, and I used it to my advantage to help me build my UI and design my UX overall. I was able to create actors such as an average user and a prospective user to help guide my building of the UX. The most apparent example of this is the demo button at the top of the page. Rather than explaining exactly how to do everything in a overbearing documentation set, hitting the button shows the quick demo. The system is rather simple, but I believe it does a good job converting novice users to more seasoned users. I was able to do this by using think out loud methods with test subjects and brainstorming ideas with key stakeholders. Overall, activity design scenarios and other elements drove a majority of my UI design.

Evaluation and Redesign

Given that there were no major example systems that I based my design off, evaluation and redesign of my system was very important. I was very lucky that I was able to get a lot of feedback from users, as well as anticipate what a user may or may not need or want from this system. Although it is a pretty simple concept, a lot went into it. For that reason, I spent a majority of my time performing formative evaluations. This is yet another major aspect to SBD that guides the developer in reformation of functionality, UI, and overall plot changes. By evaluating current problem and activity scenarios, I was able to spur ideas for new ones. For example, I initially looked at the issue of the user being unable to understand where the data came from, and therefore the system would likely seem less viable. I analyzed this problem scenario and formed an activity scenario with the common metaphor of a citation. People understand citations, which gives literature credibility. Especially in the modern world, people rarely tend to take things at face value. Therefore, I chose to “sight” my data sources with images and links to the listings that were analyzed.

(Cited Output)

I had three phases of prototyping that fell in order of fidelity. The first that I created was a non-GUI version of the system that did not have real back-end function. The goal for this was to spur ideas and to get a quick and dirty prototype in front of real users. This would allow me to get their feedback as well as understand what users are looking for in a system. The results that I got from this stage were very helpful. I was able to see that user understanding of input is crucial in terms of where to input and what functions they need to do in order to get a result. The SBD concept that applied most here was the fact that users are motivated to produce results, not to understand a system. Therefore, I made sure that input was a lot simpler in the second prototype.

(Prototype 2 Search Bar)

I continued my empirical evaluation in the second prototype as well. This system was a lot more functional with actual back-end function. It was easier for the user to understand with inputs, but it did come with its issues as well. The second prototype helped me to better understand my users with a second group and demographic survey. I was able to determine that my average pool of users would likely have some computer knowledge given that they were savvy enough to open an online store. I made the generalization that motivated entrepreneurs generally have some sort of technological proficiency. Additionally, I was also met with some UI and back-end issues that were prompted by the “think-out-loud” form of evaluation. Almost every user in the second test wondered where the data was coming from and what exactly they should put into the search bar that was labeled “search”. This also showed me that users are more motivated by results given that I provided a prompt at the top to put a keyword in. From this, I determined that I would cite my sources by giving links to analyzed listings and I would have the system give a simple demo to start the user on the right track.

The final part of the evaluation that I used was summative. Summative is an SBD evaluation form that involves evaluating if the system performs its overall function. Although this is usually part of the final evaluation, I used it intermittently to determine if more micro level functions were up to par. For this, I mainly evaluated speed and understandability of the design. In terms of speed, I initially had the front-end make most of the server calls. However, I found that the back-end was far more powerful which would enhance the user experience given that results would certainly be at the forefront. Additionally, I reformed the structure of the output pane to make sure that it is understandable with some explicit structure given that this ultimately leads to the system goal (which is summative). The overall system goal that I evaluated was whether or not the system provides a viable solution to search for optimized tags based on a keyword. I believe the demo will show that this is in fact achieved.

Reflection On The Product

I believe that the final design of the system fully addresses the problems that I identified during the planning and development process. The system quickly and efficiently take the input of the keywords and outputs the intended goal of the optimized tags. This will allow the user to implement these tags in their own listings and receive the benefit of more page views and sales. After doing extensive research on the topic of tags, I have determined that these are a major factor in listing success, which provides more legitimacy to the system. Over the course of development, the summative system goal did not really change at all. The goal was always to provide optimized tags to the user based on a keyword input.

I also believe that many of the techniques that I described in the UI design section ratify the fact that the system is very user friendly and basically shows the user itself how to functionally get the intended result. I did this by having a demo where the user can repeatedly see how to use the system and using JavaScript scrolling functions to guide the user to where they ultimately need to go. In the real world, I believe that this system would be easy to understand given the virtual metaphor of the search bar and the strong incentive to achieve the user goal. The user goal is to optimize keywords which is a subgoal of increasing revenue. Given that there is a monetary incentive, it is better to have a system that shows itself how to be used.

This system has minimal barriers to adoption in my opinion. From what I have researched, there are no systems out there that are even close to being the same as it. I was surprised by this initially, but I was happy to be the first one to create something like this. Again, I think that the monetary incentive will be a major driving force in adoption since tags are becoming more apparent in listings. Etsy even emphasizes the use of tags now because it is beneficial for their algorithms. The only major issue that I see with this system is scalability given the 10,000 request per day rate limit. However, there is the simple solution of requesting a greater limit or authenticating users. However, authenticating users could produce a privacy issue.

Reflection On The Process

I used SBD as a major driving force of my systems development. As we have learned in class, SBD has many benefits that help developers to produce viable systems that fit user needs. For example, using SBD gave me the benefit of fewer design changes, less need for user training, and most likely an increased user loyalty. This is known as cost-justifying usability, and it is one of the major benefits to using SBD. By conducting my empirical evaluations, I was able to see real users try my system. This helped me to gain valuable information not only on the present system, but the entire goal overall. Additionally, I was able to use minimalist design and documentation to help the user to better understand functionality without the need of a massive user documentation manual.

I think scenario writing and resulting claims really helped me to design my UX and overall system. These pushed me to look through the lens of various stakeholders and pose problems for me to solve. From there I was able to produce scenarios and develop metaphors that users would be able to leverage for understanding. The major examples of this were the search bar and citations that I previously mentioned. For UI design I was able to use hierarchical task evaluation (HTA) to break down tasks so that I could understand the full user experience and understanding. This gave me a wholistic view of what needed to be implemented.

The greatest part of SBD that I leveraged was formative evaluation. This is the concept that stuck out to me most in class since it guides iterative development. Obviously, for a system that has never been put into production before, this is a crucial concept for its evaluation. I had many forms of my prototypes, and this concept allowed me to draw in other aspects like problem and activity scenarios to create a system that is really well rounded. I would definitely say that I would use SBD to tackle upcoming tasks at my job. I spent the past summer working on Wall Street with a number of different tools (even some I made myself). At one point I could not understand why others were unable to understand my UI, and now I have a way to solve this with SBD.
